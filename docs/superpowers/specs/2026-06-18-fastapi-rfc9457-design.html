Implementation Spec

fastapi-rfc9457

Typed, batteries-included RFC 9457 “Problem Details for HTTP APIs” integration for the latest FastAPI & Pydantic v2.

Status: approved for planning FastAPI ≥ 0.137 Pydantic ≥ 2.13 Python ≥ 3.11 hatch-vcs

1 · Summary & positioning

fastapi-rfc9457 is a reusable package that makes a FastAPI app emit RFC 9457 application/problem+json error bodies with one call, and lets you declare each error condition as a small typed class whose extension members are validated and documented.

The market today is a two-horse race, and neither horse fills the whole field:

LibraryMaint.Pydantic v2Typed, validated extension membersAuto per-route OpenAPI (problem+json)Client-side parsing
fastapi-problem (NRWLDev)activeno (multidict)no (kwargs)partialno
fastapi-problem-details (g0di)light, pre-1.0yesno (unvalidated)best of the twono
fastapi-rfc9457 (this)newyesyesyes (validated)yes

FastAPI itself lists “Errors based on RFC 9457” on its roadmap (issue #10370, not started). Our differentiated whitespace: typed + validated extension members that round-trip through serialization, OpenAPI, and the client; automatic per-route OpenAPI registration under the correct media type; and a client-side parser — wrapped in a one-call add_problem_details(app) plus an optional composable lifespan.

2 · Goals & non-goals

Goals

  • One-call wiring: add_problem_details(app) registers handlers + OpenAPI registration (fastapi-pagination ergonomics).
  • Declare a problem type as a small class; raise it directly; typed extension members validated at the raise site.
  • Uniform application/problem+json across all errors: our problems, RequestValidationError, HTTPException/StarletteHTTPException, and unhandled Exception.
  • Structured 422 bodies that preserve field + list-index mapping (no flattening).
  • Accurate per-route OpenAPI under application/problem+json, derived from the types a route declares.
  • Dereferenceable type URIs via an optional docs router.
  • Client-side parsing of problem+json back into typed problems.
  • Modern foundations only: lifespan, Annotated, Pydantic v2; hatch-vcs packaging.

Non-goals (YAGNI)

  • No framework-agnostic core for Starlette/Litestar/Flask in v1 — FastAPI-first. (A pure core can be factored out later if demand appears.)
  • No i18n/localization of title/detail.
  • No problem-type registry persistence or remote type catalogs.
  • No middleware-based interception — exception handlers only (the field consensus; middleware is the abandoned pattern).

3 · Key decisions (empirically verified)

Each of these was checked against the real FastAPI 0.137 / Pydantic 2.13 in the project venv, not assumed.

DecisionEvidence
Problem types are frozen Pydantic dataclasses that subclass Exception — raised directly (raise OutOfCredit(...)). class Problem(BaseModel, Exception) fails at runtime: “multiple bases have instance lay-out conflict.” A pydantic @dataclass(kw_only=True) subclassing Exception raises cleanly and produces a JSON schema via TypeAdapter.
title/status are ClassVars; extension members are real fields. pyright flags status = "nope" as reportAssignmentType; flags a missing/mistyped extension member at construction (reportCallIssue / reportArgumentType); reveals .balance as int.
The terse class P(Problem, title=..., status=...) form is rejected. Pydantic’s metaclass owns class keyword args; title collides with Pydantic’s reserved config key and the real type error drowns in ~48 spurious ones; it also raises TypeError at runtime.
frozen=True is the default for problem classes. A frozen pydantic-dataclass exception raises fine (twice), raise … from … sets __cause__, __traceback__ is populated, and post-construction mutation raises FrozenInstanceError.
Accurate problem+json OpenAPI needs the one-time app.openapi component registration. In a live OpenAPI dump: {"model": X} documents application/json only; {"model": X, "content": {problem+json:{}}} yields a messy dual media type with the schema on the JSON one; the clean single problem+json + $ref needs the component registered.

5 · Core models

Two models, cleanly separated: an authoring surface and a wire surface.

5.1 Wire model — ProblemDetail

The canonical serialized shape, and what generic client parsing produces.

class ProblemDetail(BaseModel):
    model_config = ConfigDict(extra="allow")     # carry unknown extension members on the wire
    type:     str = "about:blank"
    title:    str
    status:   int
    detail:   str | None = None
    instance: str | None = None

Serialized with model_dump_json(exclude_none=True) — verified to drop unset standard members while keeping extension members — and emitted with media type application/problem+json.

5.2 Authoring surface — Problem

A frozen Pydantic dataclass that subclasses Exception. title/status are ClassVars declared on the base (so the type checker validates subclass assignments); extension members are real typed fields.

from fastapi_rfc9457 import Problem

class OutOfCredit(Problem):
    """The account does not have enough credit."""   # docstring → OpenAPI description + docs page
    title  = "Out of Credit"                          # checked against ClassVar[str] on the base
    status = 403                                       # checked against ClassVar[int] on the base
    balance:  int                                      # typed extension members (RFC 9457 §3.2)
    accounts: list[str]

# raise it directly — extension members are statically type-checked:
raise OutOfCredit(detail="Not enough credit.", balance=30, accounts=["/acct/12"])
Verified statically (pyright): OutOfCredit(detail="x") → “Arguments missing for parameters balance, accounts”; balance="lots" → wrong-type error; status = "nope" → assignment-type error; .balance reveals as int.

Subclasses require no decorator: Problem carries the dataclass machinery and is marked with @dataclass_transform so the checker treats every subclass as a (kw-only, frozen) dataclass, while __init_subclass__ registers the type. The base, schematically:

@dataclass_transform(kw_only_default=True, frozen_default=True)
class _ProblemMeta(type): ...        # or a re-exported decorator; mechanism pinned in §18

@pydantic.dataclasses.dataclass(kw_only=True, frozen=True)
class Problem(Exception, metaclass=_ProblemMeta):
    title:    ClassVar[str]
    status:   ClassVar[int]
    type:     ClassVar[str | None] = None   # optional per-type override of the derived type URI
    detail:   str | None = None
    instance: str | None = None

    def __init_subclass__(cls, **kw): ...   # apply dataclass processing + register in the type registry
The exact zero-boilerplate mechanism (custom metaclass vs. a re-exported @problem decorator) is the single open implementation item — it ships only after a pyright test proves subclasses keep full field checking. See §18.

6 · Type-URI strategy

  • Default type = {config.base_url}/{kebab-id}, where kebab-id is the class name minus a trailing Problem/Error, lower-kebab-cased → OutOfCredit ⇒ /problems/out-of-credit.
  • config.base_url defaults to /problems; may be an absolute URL (https://api.example.com/problems) or overridden per type via the type ClassVar.
  • Because the package ships a docs router, the default type is dereferenceable (the RFC’s intent) rather than about:blank.

7.1 Wiring example

from fastapi import FastAPI
from fastapi_rfc9457 import add_problem_details, Problem, problems

class PostNotFound(Problem):
    """The requested post does not exist."""
    title  = "Not Found"
    status = 404

app = FastAPI()
add_problem_details(app)          # handlers + OpenAPI registration, idempotent

@app.get("/posts/{post_id}", responses=problems(PostNotFound))
async def get_post(post_id: str) -> PostResponse:
    post = await store.get(post_id)
    if post is None:
        raise PostNotFound(detail=f"Post {post_id} not found")
    return post

8 · Exception handling

add_problem_details registers four handlers. Every error on the wire becomes application/problem+json.

flowchart LR A[raise Problem subclass] --> H1[problem_handler] B[RequestValidationError] --> H2[validation_handler] C["HTTPException / StarletteHTTPException"] --> H3[http_handler] D[unhandled Exception] --> H4[unhandled_handler] H1 --> W[build fresh ProblemDetail per request] H2 --> W H3 --> W H4 --> W W --> R["JSONResponse · application/problem+json"]
HandlerCatchesProduces
problem_handlerProblem (so every authored type)Its wire ProblemDetail; auto-fills instance from the request path if unset.
validation_handlerRequestValidationErrorValidationProblem (422) with a typed errors[] extension — see §9.
http_handlerStarletteHTTPException (also covers FastAPI’s HTTPException, a subclass)A ProblemDetail with title from the HTTP status phrase. Catches errors raised by Starlette internals and third-party middleware too.
unhandled_handlerExceptionInternalServerError (500), honoring strip_debug.
Documented caveats (Starlette behavior): under app.debug=True Starlette renders a traceback page instead of invoking the 500 handler, so the problem+json 500 path isn’t exercised in debug; and Starlette re-raises the unhandled exception after the handler runs (for logging) — both expected, both noted in the README.

8.1 The fresh-wire-model invariant

Handlers never mutate the carried exception. Each request materializes a brand-new ProblemDetail from the problem’s ClassVar constants + dataclass fields, and resolves instance on that fresh model. This makes module-level/memoized problems and concurrent requests safe by construction (see §10).

9 · Validation → errors[] (no flattening)

FastAPI’s default 422 is {"detail": [{"loc":…,"msg":…}]}. Flattening that into a single string loses the field/index mapping (e.g. which item in a list body failed). We carry the structured errors as a typed RFC 9457 extension member — dogfooding the typed-extensions feature.

class InvalidParam(BaseModel):
    loc:     list[str | int]    # faithful FastAPI location, e.g. ["body", 0, "task_name"]
    pointer: str | None = None  # RFC 6901 convenience, e.g. "/body/0/task_name"
    detail:  str                # the pydantic message
    type:    str                # pydantic error code, e.g. "missing"
    # `input` (the offending value) is attached only when config.strip_debug is False

class ValidationProblem(Problem):
    """The request failed validation."""
    title  = "Unprocessable Content"
    status = 422
    errors: list[InvalidParam]

The handler maps exc.errors() 1:1 (no join); detail is a short summary. A list body that fails on items 0 and 2:

{
  "type": "/problems/validation", "title": "Unprocessable Content", "status": 422,
  "detail": "Request validation failed (2 errors).",
  "instance": "/v1/jobs",
  "errors": [
    {"loc": ["body", 0, "task_name"], "pointer": "/body/0/task_name", "detail": "Field required", "type": "missing"},
    {"loc": ["body", 2, "task_name"], "pointer": "/body/2/task_name", "detail": "Field required", "type": "missing"}
  ]
}
  • loc is authoritative (preserves the list index); pointer is an RFC 6901 convenience. We use loc/pointer rather than RFC 7807’s {name, reason} precisely because name can’t express an array index.
  • input is gated by strip_debug to avoid leaking submitted values in production.
  • Covers query/path/header errors uniformly (loc[0] ∈ {body,query,path,header,cookie}).
Adopter-visible change: this replaces FastAPI’s default 422 application/json body with application/problem+json — intended, since the whole point is a uniform surface. Called out in the README migration notes.

10 · Immutability & shared-state safety

A real-world review flagged in-place mutation of instance bleeding across requests when a problem is memoized. We close that bug class two ways:

  1. Fresh-wire-model invariant (§8.1): the carried Problem is read-only input; the handler builds a new ProblemDetail per request and sets instance there. No path bleed, no race, even if one problem instance is shared across concurrent requests.
  2. frozen=True default: post-construction mutation raises FrozenInstanceError (verified), so even buggy code can’t mutate a carried problem. This makes the module-level-constant pattern recommended:
POST_NOT_FOUND = PostNotFound()   # define once, reuse everywhere — frozen makes it correct
...
raise POST_NOT_FOUND              # each request still gets its own resolved `instance`

Per-raise detail/instance are constructor arguments, so nothing is lost. strip_debug likewise redacts into a new wire model, never the source.

11 · OpenAPI integration

11.1 Why add_problem_details registers components

A live OpenAPI dump settles the “do we need to touch app.openapi?” question:

responses= entryDocumented media type(s)Component registered?
{"model": ProblemDetail}application/json onlyyes
{"model": …, "content": {problem+json: {}}}dual: problem+json (empty) + application/json (the $ref)yes
{"content": {problem+json: {"schema": {"$ref": …}}}} ✅application/problem+json only, with $refno (needs registration)

Only the hand-written $ref form documents the correct, single media type — and it needs its component registered once. So problems() emits the $ref form, and add_problem_details wraps app.openapi to inject ProblemDetail + every registered type’s schema into components.schemas (with lazy app-metadata binding, faststream-style). This is the FastAPI-endorsed Extending OpenAPI pattern and is idempotent.

Fallback for adopters who want zero app.openapi wrapping: use model-based responses documented as application/json. We do not default to that — accuracy matters for an RFC 9457 package.

11.2 problems(*types) — composition

Returns a plain dict, so it spreads into any responses=:

responses={**problems(OutOfCredit, PostNotFound), 200: {...}, 503: {"model": SomeOtherLibModel}}
  • Different statuses → merge cleanly.
  • Two of our types at the same status → merged into one entry: content: {problem+json: {schema: {oneOf: [{$ref: A}, {$ref: B}]}}}, with descriptions joined. (An improvement over the skill, which kept only one $ref.)
  • Collision with a non-ours response at the same status → last-writer-wins, like any dict; the caller merges by hand. problems() stays transparent so they can.

12 · Client-side parsing

No competitor offers this. For services/SDKs that consume a problem+json API:

from fastapi_rfc9457 import parse_problem, raise_for_problem

problem = parse_problem(resp)          # -> ProblemDetail (generic; extras preserved)
                                       #    or the typed Problem subclass if `type` is registered
raise_for_problem(resp)                # raises the mapped Problem / ProblemError if it's a problem response

import httpx
client = httpx.Client(event_hooks={"response": [httpx_raise_hook()]})  # auto-raise on problem responses
  • parse_problem accepts a response object, dict, or bytes.
  • Registry-aware: when type matches a registered type, returns that typed instance (extension members restored typed); otherwise a generic ProblemDetail.
  • The type→class registry is the same one populated by __init_subclass__.

13 · Type-doc router

build_docs_router(config) returns an APIRouter serving GET {base_url}/{kebab-id} for each registered type — a human-readable page (HTML + JSON via content negotiation) showing title, status, the docstring, and the extension-member schema. Makes type URIs resolve to documentation per the RFC’s intent. Path is configurable; include_in_schema toggle so the doc routes don’t clutter the main API schema.

14 · Configuration — ProblemConfig

class ProblemConfig(BaseModel):
    base_url:              str  = "/problems"   # type-URI prefix; may be absolute
    include_docs_router:   bool = True
    strip_debug:           bool = False         # hide detail/extras/input on 500s and validation in prod
    instance_from_request: bool = True          # auto-fill `instance` from the request path
    validation_type:       str  = "validation"  # kebab id for ValidationProblem
    validation_status:     int  = 422

Passed to add_problem_details(app, config) and build_docs_router(config).

15 · Lifespan composition (the faststream pattern)

Handlers and the app.openapi wrap must be eager, so add_problem_details does those. The optional problem_details_lifespan adds startup work: validate the registry (fail-fast on duplicate type URIs or missing title/status), bind app metadata, and mount the docs router. It composes with the user’s own lifespan exactly as faststream does — user lifespan outer, ours nested:

from contextlib import asynccontextmanager
from fastapi import FastAPI
from fastapi_rfc9457 import add_problem_details, problem_details_lifespan

@asynccontextmanager
async def lifespan(app: FastAPI):
    async with problem_details_lifespan(app):   # nests inside your own startup/shutdown
        # ... your resources ...
        yield

app = FastAPI(lifespan=lifespan)
add_problem_details(app)

Use either or both: add_problem_details alone is sufficient for most apps; add the lifespan when you want startup validation / docs mounting.

17 · Testing strategy

TDD throughout (write the failing test first). Two test tiers:

Runtime — TestClient / httpx

  • Each handler: status code, content-type: application/problem+json, body shape.
  • Validation guard: a multi-error list body asserting per-item loc/field/type in errors[] — the regression the review described must fail CI.
  • State-safety guard: raise one shared module-level frozen problem from two requests with different paths; assert each response carries its own instance and the source object is unchanged.
  • HTTPException/StarletteHTTPException interception; unhandled-Exception → 500; strip_debug on/off.
  • OpenAPI: components.schemas contains ProblemDetail + typed types; route responses reference them under problem+json; oneOf merge at a shared status; no dangling $ref.
  • Docs router pages; client round-trip (typed extras restored); add_problem_details idempotency warning.

Static — pyright suite (tests/static/)

  • Files that must type-check, and files that must produce specific errors (missing/mistyped extension member, wrong status type), asserted via reveal_type / expected-error markers — locking in the headline typing guarantee. Run under uv run pyright in CI.

18 · Risks & open items

ItemPlan
Zero-boilerplate subclass mechanism (metaclass vs. @problem decorator) while keeping full pyright field checking. First task in the plan is a spike: a tests/static/ file proving a decorator-free subclass still type-checks. If the metaclass route is fiddly, ship a one-line @problem decorator as the documented form. Both are known-viable; the dataclass+Exception+frozen+ClassVar core is already verified.
Per-type wire schema generation (type/title/status as consts + typed extras + detail/instance) for OpenAPI. Build via TypeAdapter/a generated ProblemDetail subclass per type; verified TypeAdapter(...).json_schema() yields the extension fields. Registered by add_problem_details.
FastAPI ships its own RFC 9457 support (roadmap #10370). Ship a stable 1.0 early; the typed-extensions + client-parsing differentiators are unlikely to be matched by a minimal first-party feature.
frozen + mutable field defaults (e.g. list) aren’t hashable. Acceptable — hashing isn’t required; frozen still blocks attribute reassignment (the actual bug class).
